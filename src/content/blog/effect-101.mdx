# Effect 101: How to Read Effect Code

People often say [Effect.ts](https://effect.website/) has a steep learning curve, and I get it. When you first see `Effect.gen`, `yield*`, `Layer`, `Runtime`, `Data.TaggedError`, and services everywhere, it can feel like Effect is trying to replace TypeScript with a secret language. But that's not the right way to read it. Effect code is still TypeScript code, the trick is just learning how to read it. Let me show you.

---

## `Effect.gen` Is Where It Clicks

This is the first piece of Effect code you should learn to read.

```ts twoslash id="start-effect-gen" collapse={1-30}
import { Effect } from "effect"

interface User {
  readonly id: string
  readonly name: string
}

interface Appointment {
  readonly id: string
  readonly userId: string
  readonly status: "booked"
}

const getUser = (): Effect.Effect<User> =>
  Effect.gen(function* () {
    return {
      id: "user_1",
      name: "John",
    }
  })

const getAppointment = (): Effect.Effect<Appointment> =>
  Effect.gen(function* () {
    return {
      id: "appointment_1",
      userId: "user_1",
      status: "booked",
    }
  })

const program = Effect.gen(function* () {
  const user = yield* getUser()
  const appointment = yield* getAppointment()

  console.log(`Loaded appointment for ${user.name}`)

  return appointment
})

Effect.runPromise(program).then(console.log)

/*
Output:
Loaded appointment for John
{ id: "appointment_1", userId: "user_1", status: "booked" }
*/
```

The mental model is simple. Read this:

```ts id="read-yield"
const user = yield * getUser()
```

exactly like this:

```ts id="read-await"
const user = await getUser()
```

Inside `Effect.gen`, the line `yield* someEffect` means:

> Run this Effect step. \
> If it succeeds, give me the success value. \
> If it fails, stop the whole program with that failure.

The Effect version above is basically the same story as this normal TypeScript:

```ts twoslash id="async-await-shape" collapse={1-22}
interface User {
  readonly id: string
  readonly name: string
}

interface Appointment {
  readonly id: string
  readonly userId: string
  readonly status: "booked"
}

const getUser = async (): Promise<User> => ({
  id: "user_1",
  name: "John",
})

const getAppointment = async (): Promise<Appointment> => ({
  id: "appointment_1",
  userId: "user_1",
  status: "booked",
})

async function program() {
  const user = await getUser()
  const appointment = await getAppointment()

  console.log(`Loaded appointment for ${user.name}`)

  return appointment
}

program().then(console.log)

/*
Output:
Loaded appointment for John
{ id: "appointment_1", userId: "user_1", status: "booked" }
*/
```

The shape is identical. Effect isn't asking you to forget `async/await`, it's giving you a typed version of the same thing, the only difference is that `yield*` works with Effect values instead of Promise values.

## Errors That Show Up in the Type

Now let's make the example more realistic. We'll model a flow that finds a user, finds their appointment, returns it, and converts any failure into a single domain error.

```ts twoslash id="concrete-error-handling" collapse={1-53}
import { Data, Effect } from "effect"

interface User {
  readonly id: string
  readonly name: string
}

interface Appointment {
  readonly id: string
  readonly userId: string
  readonly status: "booked" | "cancelled"
}

class UserNotFound extends Data.TaggedError("UserNotFound")<{
  readonly userId: string
}> {}

class AppointmentNotFound extends Data.TaggedError("AppointmentNotFound")<{
  readonly userId: string
}> {}

class BookingLookupError extends Data.TaggedError("BookingLookupError")<{
  readonly message: string
  readonly cause: UserNotFound | AppointmentNotFound
}> {}

const getUser = (userId: string): Effect.Effect<User, UserNotFound> =>
  Effect.gen(function* () {
    if (userId !== "user_1") {
      return yield* new UserNotFound({ userId })
    }

    return {
      id: "user_1",
      name: "John",
    }
  })

const getAppointment = (
  userId: string,
): Effect.Effect<Appointment, AppointmentNotFound> =>
  Effect.gen(function* () {
    if (userId !== "user_1") {
      return yield* new AppointmentNotFound({ userId })
    }

    return {
      id: "appointment_1",
      userId,
      status: "booked",
    }
  })

const getUserAppointment = (userId: string) =>
  Effect.gen(function* () {
    const user = yield* getUser(userId)
    const appointment = yield* getAppointment(user.id)

    return appointment
  }).pipe(
    Effect.catchTags({
      UserNotFound: (error) =>
        new BookingLookupError({
          message: `Cannot find user ${error.userId}`,
          cause: error,
        }),
      AppointmentNotFound: (error) =>
        new BookingLookupError({
          message: `Cannot find appointment for user ${error.userId}`,
          cause: error,
        }),
    }),
  )

const program = getUserAppointment("user_1")
//    ^?

Effect.runPromise(program).then(console.log)

/*
Output:
{ id: "appointment_1", userId: "user_1", status: "booked" }
*/
```

Notice that we never annotated the return type, we let Effect infer it. Hover `program` in your editor and you get the whole story: `Effect<Appointment, BookingLookupError, never>`.
Read it as:

> This program succeeds with an `Appointment`, \
> may fail with `BookingLookupError`, \
> and needs no dependencies.

For contrast, here's the same idea written the old way, with `try/catch`:

```ts twoslash id="normal-try-catch-throw" collapse={1-55}
interface User {
  readonly id: string
  readonly name: string
}

interface Appointment {
  readonly id: string
  readonly userId: string
  readonly status: "booked" | "cancelled"
}

class UserNotFound extends Error {
  constructor(readonly userId: string) {
    super(`User not found: ${userId}`)
  }
}

class AppointmentNotFound extends Error {
  constructor(readonly userId: string) {
    super(`Appointment not found for user: ${userId}`)
  }
}

class BookingLookupError extends Error {
  constructor(
    readonly message: string,
    readonly cause: unknown,
  ) {
    super(message)
  }
}

async function getUser(userId: string): Promise<User> {
  if (userId !== "user_1") {
    throw new UserNotFound(userId)
  }

  return {
    id: "user_1",
    name: "John",
  }
}

async function getAppointment(userId: string): Promise<Appointment> {
  if (userId !== "user_1") {
    throw new AppointmentNotFound(userId)
  }

  return {
    id: "appointment_1",
    userId,
    status: "booked",
  }
}

async function getUserAppointment(userId: string): Promise<Appointment> {
  try {
    const user = await getUser(userId)
    const appointment = await getAppointment(user.id)

    return appointment
  } catch (error) {
    if (error instanceof UserNotFound) {
      throw new BookingLookupError(`Cannot find user ${error.userId}`, error)
    }
    if (error instanceof AppointmentNotFound) {
      throw new BookingLookupError(
        `Cannot find appointment for user ${error.userId}`,
        error,
      )
    }
    throw error
  }
}

const promise = getUserAppointment("user_1")
//    ^?

promise.then(console.log)

/*
Output:
{ id: "appointment_1", userId: "user_1", status: "booked" }
*/
```

This is the real value of Effect. Where normal TypeScript would only say: `Promise<Appointment>` Effect tells you more: `Effect<Appointment, BookingLookupError, never>`.
The failure isn't hidden, it's part of the type.

## Wait, Why Does `Effect.gen` Take No Arguments?

This confuses a lot of beginners. You usually write:

```ts id="effect-gen-no-args-shape"
const program = Effect.gen(function* () {
  // ...
})
```

Not:

```ts id="effect-gen-with-args-wrong-shape"
const program = Effect.gen(function* (userId: string) {
  // ...
})
```

Why?

Because `Effect.gen` is where you only describe **computation**.
It is not where you receive business arguments.
A useful way to think about it is:

> An `Effect` is a lazy computation. \
> It describes what should happen later. \
> It does not run immediately.

So this:

```ts id="effect-gen-no-args"
const program = Effect.gen(function* () {
  return 42
})
```

means:

> Here is a lazy program. \
> When someone runs it later, it will produce `42`.

`program` is not a function.

It is already an `Effect`.

It is a **result value** that **has not run**.

Think of `Effect.Effect<Success, Error>` as `Promise<Result<Success, Error>>` but the promise has not run yet.

That is why `Effect.gen` does not need arguments here. There is nothing to pass in at runtime. The program already knows everything it needs.

But most real business logic does need inputs.

For example:

```ts id="business-operation-shape"
greet(name)
getUser(userId)
createAppointment(input)
```

In those cases, the normal function receives the arguments, and then returns an `Effect`.

```ts twoslash id="normal-function-returning-effect"
import { Effect } from "effect"

const greet = (name: string) =>
  Effect.gen(function* () {
    return `Hello, ${name}`
  })

greet
// ^?

const program = greet("John")
```

The outer function receives the argument.

```ts id="outer-function-receives-arguments"
const greet = (name: string) => ...
```

The inner `Effect.gen` describes the lazy computation.

```ts id="inner-effect-gen-describes-computation"
Effect.gen(function* () {
  return `Hello, ${name}`
})
```

So when you call `const program = greet("John")` you are not saying "**Run this now**."

You are saying "Create a lazy Effect that already remembers `name = "John"`".

The value `"John"` is captured by the function closure. Then the Effect can be run later

```ts id="run-program-later"
Effect.runPromise(program)
```

This is different from `async` functions.

```ts twoslash id="async-starts-immediately"
const greet = async (name: string) => {
  return `Hello, ${name}`
}

const promise = greet("John")

/*
Output:
Hello, John
*/
```

With `async`, calling the function creates and starts a `Promise` eagerly.

## `Effect.fn`: Give Your Functions a Name

`Effect.fn` creates named functions that return Effects. Its arguments should be normal business inputs like `userId`, `appointmentId`, `order`, `paymentInput`, `command`, or `query`.

```ts twoslash id="effect-fn-example" collapse={1-42}
import { Data, Effect } from "effect"

interface User {
  readonly id: string
  readonly name: string
}

interface Appointment {
  readonly id: string
  readonly userId: string
}

class UserNotFound extends Data.TaggedError("UserNotFound")<{
  readonly userId: string
}> {}

class AppointmentNotFound extends Data.TaggedError("AppointmentNotFound")<{
  readonly userId: string
}> {}

const getUser = Effect.fn("getUser")(function* (userId: string) {
  if (userId !== "user_1") {
    return yield* new UserNotFound({ userId })
  }
  const user: User = {
    id: userId,
    name: "John",
  }
  return user
})

const getAppointment = Effect.fn("getAppointment")(function* (userId: string) {
  if (userId !== "user_1") {
    return yield* new AppointmentNotFound({ userId })
  }
  const appointment: Appointment = {
    id: "appointment_1",
    userId,
  }
  return appointment
})

const getAppointmentForUser = Effect.fn("getAppointmentForUser")(function* (
  userId: string,
) {
  const user = yield* getUser(userId)
  const appointment = yield* getAppointment(user.id)

  return appointment
})

const program = getAppointmentForUser("user_1")

Effect.runPromise(program).then(console.log)

/*
Output:
{ id: "appointment_1", userId: "user_1" }
*/
```

Read this `Effect.fn("getAppointmentForUser")` as

> Define a named function called `getAppointmentForUser` that returns an Effect

The name is useful for tracing, debugging, and observability, the argument is the business input, and the function still returns a lazy Effect program.

## Three Channels to Keep in Mind

Effect tracks different kinds of outcomes. For beginners, think of three channels:

| Channel | Meaning                  | Example                                               |
| ------- | ------------------------ | ----------------------------------------------------- |
| Success | The program succeeded    | `Effect.succeed(value)`                               |
| Failure | Expected/tracked error   | `Effect.fail(error)` or `yield* new TaggedError(...)` |
| Defect  | Unexpected/untracked bug | `Effect.die(error)` or unhandled throw                |

Failure means "this is a normal error that can happen, and I want it tracked." Defect means "this should not happen, or I didn't handle it as a typed failure."

### Failure: Errors You Saw Coming

Use `Data.TaggedError` for expected domain errors. Inside `Effect.gen`, tagged errors are yieldable.

```ts twoslash id="failure-channel-example"
import { Data, Effect } from "effect"

class InsufficientFunds extends Data.TaggedError("InsufficientFunds")<{
  readonly balance: number
  readonly required: number
}> {}

const chargeCard = Effect.fn("chargeCard")(function* (balance: number) {
  const required = 100

  if (balance < required) {
    return yield* new InsufficientFunds({ balance, required })
  }

  return `Payment success: charged ${required}, balance ${balance}`
})

const program = (balance: number) =>
  chargeCard(balance).pipe(
    Effect.catchAll((error) =>
      Effect.succeed(
        `Payment failed: required ${error.required}, balance ${error.balance}`,
      ),
    ),
  )

Effect.runPromise(program(50)).then(console.log)
Effect.runPromise(program(150)).then(console.log)

/*
Output:
Payment failed: required 100, balance 50
Payment success: charged 100, balance 150
*/
```

The failure is tracked: before `catchAll`, the program may fail with `InsufficientFunds`, and after `catchAll`, the error is handled, so a low balance becomes a failure message while a sufficient one succeeds.

### Defect: Errors You Didn't

Use `Effect.die` for bugs or impossible states.

```ts twoslash id="defect-channel-example"
import { Effect } from "effect"

const processPayment = Effect.fn("processPayment")(function* (balance: number) {
  if (balance < 0) {
    return yield* Effect.die("Invariant violated: balance is negative")
  }

  return {
    paid: true,
  }
})

const program = processPayment(-1)

Effect.runPromiseExit(program).then(console.log)

/*
Output:
Exit.fail(...Cause.die...)
*/
```

A defect isn't a normal business failure, it means something is wrong with the program itself, things like an impossible state, corrupted data, a programming mistake, an unexpected null, or a broken invariant. Importantly, `Effect.catchAll(...)` catches expected failures but does not handle defects like normal business errors. That's a good thing, you usually don't want to recover from bugs inside your domain flow.

## `Effect.gen` for Logic, `.pipe()` for Everything Around It

For beginners, this is the style I recommend, use `Effect.gen` for the main story and `.pipe()` for the behavior around it:

```ts id="gen-for-logic-pipe-for-composition-shape"
Effect.gen(function* () {
 ... // business logic lives here
})
.pipe(
 ... // composition happens here
)
```

Here's a fuller example:

```ts id="gen-for-logic-pipe-for-composition"
const fetchUserPosts = (userId: string) =>
  Effect.gen(function* () {
    const cacheKey = `posts:${userId}`
    const cached = yield* cacheGet(cacheKey)
    if (cached) {
      return cached
    }
    const posts = yield* findPostsByUser(userId)
    yield* cacheSet(cacheKey, posts)
    return posts
  }).pipe(
    Effect.withSpan("fetch_user_posts"),
    Effect.retry(retryPolicy),
    Effect.catchTag("DatabaseError", () => Effect.succeed([])),
  )
```

With the setup collapsed, the shape is clear: the business logic lives inside `Effect.gen`, and the operational behavior (`withSpan`, `retry`, `catchTag`) lives inside `.pipe()`. That separation is what makes the code easy to read.

### Don't Pipe Everything Just Because You Can

This is perfectly legal, but often harder to read:

```ts id="too-much-pipe"
Effect.succeed(order).pipe(
  Effect.andThen(validateOrder),
  Effect.andThen(calculateTotals),
  Effect.andThen(applyDiscounts),
  Effect.andThen(processPayment),
  Effect.andThen(sendConfirmation),
)
```

It isn't wrong, but the moment you add conditionals, intermediate names, logs, or branching, this style becomes hard to scan. For sequential business logic, prefer this:

```ts id="prefer-gen-over-too-much-pipe"
const program = Effect.gen(function* () {
  const validated = yield* validateOrder(order)
  const withTotals = yield* calculateTotals(validated)
  const discounted = yield* applyDiscounts(withTotals)
  const payment = yield* processPayment(discounted)
  return yield* sendConfirmation(payment)
})
```

Here's my beginner decision matrix:

| Situation                            | Prefer       |
| ------------------------------------ | ------------ |
| Injecting or retrieving dependencies | `Effect.gen` |
| Conditional logic                    | `Effect.gen` |
| Sequential operations                | `Effect.gen` |
| Error handling                       | `.pipe()`    |
| Adding tracing                       | `.pipe()`    |
| Adding retry/timeout                 | `.pipe()`    |
| Layer building                       | `.pipe()`    |
| Simple transforms                    | `.pipe()`    |

## Dependencies Without the Plumbing

Now we can talk about dependencies. In normal TypeScript, dependencies are usually passed as arguments or constructor parameters.

```ts id="normal-di"
class UserService {
  constructor(private readonly database: Database) {}
}
```

In Effect, dependencies can live in the requirement channel, so you don't have to manually thread them through every function. Here's a small example.

```ts twoslash id="dependency-injection-requirement-channel" collapse={1-30}
import { Data, Effect } from "effect"

interface User {
  readonly id: string
  readonly name: string
}

class UserNotFound extends Data.TaggedError("UserNotFound")<{
  readonly userId: string
}> {}

class UserService extends Effect.Service<UserService>()("UserService", {
  effect: Effect.sync(() => {
    const getById = Effect.fn("getById")(function* (userId: string) {
      if (userId !== "user_1") {
        return yield* new UserNotFound({ userId })
      }

      const user: User = {
        id: userId,
        name: "John",
      }

      return user
    })

    return { getById }
  }),
}) {}

const program = Effect.gen(function* () {
  const userService = yield* UserService
  const user = yield* userService.getById("user_1")

  return user.name
})

program
// ^?

const runnable = program.pipe(Effect.provide(UserService.Default))

Effect.runPromise(runnable).then(console.log)

/*
Output:
John
*/
```

Before providing the service, the program needs `UserService`. After this:

```ts id="provide-user-service"
program.pipe(Effect.provide(UserService.Default))
```

the requirement is gone and the program is ready to run. That's dependency injection in Effect, the dependency isn't passed as a function argument, it's tracked by Effect itself.

## Wiring It Up Before You Run

You can't run a program that still needs requirements, so you provide them first. For small examples, this is fine:

```ts id="effect-provide-small-example"
program.pipe(Effect.provide(UserService.Default))
```

For real applications, prefer creating a runtime once.

```ts twoslash id="managed-runtime-example" collapse={1-34}
import { Data, Effect, ManagedRuntime } from "effect"

interface User {
  readonly id: string
  readonly name: string
}

class UserNotFound extends Data.TaggedError("UserNotFound")<{
  readonly userId: string
}> {}

class UserService extends Effect.Service<UserService>()("UserService", {
  accessors: true,
  effect: Effect.sync(() => {
    const getById = Effect.fn("getById")(function* (userId: string) {
      if (userId !== "user_1") {
        return yield* new UserNotFound({ userId })
      }

      const user: User = {
        id: userId,
        name: "John",
      }

      return user
    })

    return { getById }
  }),
}) {}

const program = Effect.gen(function* () {
  return yield* UserService.getById("user_1")
})

const runtime = ManagedRuntime.make(UserService.Default)

runtime.runPromise(program).then(console.log)

/*
Output:
{ id: "user_1", name: "John" }
*/
```

The common app pattern is to build your services and layers once, create a `ManagedRuntime` once, and then use that runtime at the edge of your app, such as an HTTP route handler, a queue worker, a CLI command, or a cron job.

> Normally, do not create a new `ManagedRuntime` for every request.

Create it once at application startup, then reuse it.

## Services Say What, Layers Say How

A service describes _what_ your app needs, and a layer describes _how_ to build it.

```ts twoslash id="services-and-layers-example" collapse={1-63}
import { Data, Effect, Layer, ManagedRuntime } from "effect"

interface User {
  readonly id: string
  readonly name: string
}

interface Appointment {
  readonly id: string
  readonly userId: string
}

class UserNotFound extends Data.TaggedError("UserNotFound")<{
  readonly userId: string
}> {}

class AppointmentNotFound extends Data.TaggedError("AppointmentNotFound")<{
  readonly userId: string
}> {}

class UserService extends Effect.Service<UserService>()("UserService", {
  accessors: true,
  effect: Effect.sync(() => {
    const getById = Effect.fn("getById")(function* (userId: string) {
      if (userId !== "user_1") {
        return yield* new UserNotFound({ userId })
      }

      const user: User = {
        id: userId,
        name: "John",
      }

      return user
    })

    return { getById }
  }),
}) {}

class AppointmentService extends Effect.Service<AppointmentService>()(
  "AppointmentService",
  {
    accessors: true,
    effect: Effect.sync(() => {
      const getByUserId = Effect.fn("getByUserId")(function* (userId: string) {
        if (userId !== "user_1") {
          return yield* new AppointmentNotFound({ userId })
        }

        const appointment: Appointment = {
          id: "appointment_1",
          userId,
        }

        return appointment
      })

      return { getByUserId }
    }),
  },
) {}

const MainLayer = Layer.mergeAll(
  UserService.Default,
  AppointmentService.Default,
)

const getAppointmentForUser = Effect.fn("getAppointmentForUser")(function* (
  userId: string,
) {
  const user = yield* UserService.getById(userId)
  const appointment = yield* AppointmentService.getByUserId(user.id)

  return appointment
})

const runtime = ManagedRuntime.make(MainLayer)

runtime.runPromise(getAppointmentForUser("user_1")).then(console.log)

/*
Output:
{ id: "appointment_1", userId: "user_1" }
*/
```

You can read the whole architecture top to bottom like this:

```txt id="architecture-read"
Effect.fn
  -> named use case

UserService / AppointmentService
  -> dependencies

Layer.mergeAll(...)
  -> dependency graph

ManagedRuntime.make(...)
  -> app runtime

runtime.runPromise(...)
  -> run at the edge
```

This is the point where Effect starts to feel like a framework, not just an error handling library.

---

## `Layer.provide` or `Effect.provide`?

Use `Effect.provide` when you're providing dependencies to a program:

```ts id="effect-provide"
program.pipe(Effect.provide(MainLayer))
```

Use `Layer.provide` when you're providing dependencies to another layer:

```ts id="layer-provide"
UserService.pipe(Layer.provide(Database))
```

A simple beginner rule:

| Situation                               | Use              |
| --------------------------------------- | ---------------- |
| I have a program and want to run it     | `Effect.provide` |
| I have a layer that needs another layer | `Layer.provide`  |

Most beginners reach for `Effect.provide` first, and you can pick up `Layer.provide` later when your dependency graph grows.

## Putting It All Together

Effect is a framework, and its code is easiest to read when you start with `Effect.gen`, which lets you write business logic just like `async/await`. From there, `yield*` is the Effect version of `await`, `.pipe()` handles composition (error handling, tracing, retry, timeout, and providing layers), `Effect.fn` creates named effectful functions with arguments, and `Data.TaggedError` defines typed domain errors. As for outcomes, the failure channel is for expected errors, the defect channel is for bugs and impossible states, and the requirement channel is dependency injection, where `Layer` builds your dependencies and `ManagedRuntime` runs your app with them.

Once that clicks, Effect becomes much less scary. It's not a secret language, it's just TypeScript with the hidden parts turned on.

## Cheatsheet: Effect vs Normal TypeScript

| Normal TypeScript                                     | Effect                                          | How to read it                                                  |
| ----------------------------------------------------- | ----------------------------------------------- | --------------------------------------------------------------- |
| `Promise<A>`                                          | `Effect<A, E, R>`                               | A lazy program that can succeed, fail, and require dependencies |
| `await promise`                                       | `yield* effect`                                 | Run this step and unwrap its success value                      |
| Business logic                                        | `Effect.gen(function* () {})`                   | Easier to read than a long pipeline                             |
| `return value`                                        | `Effect.succeed(value)`                         | Create a successful Effect                                      |
| `throw error` for expected error                      | `yield* new TaggedError(...)`                   | Stop with a typed expected failure inside `Effect.gen`          |
| `throw error` for expected error outside `Effect.gen` | `Effect.fail(error)`                            | Create a failed Effect directly                                 |
| `throw error` for bug                                 | `Effect.die(error)`                             | Mark an impossible or unexpected defect                         |
| `try/catch`                                           | `Effect.catchAll`, `Effect.catchTag`            | Handle expected typed failures                                  |
| `catch (e) if e instanceof X`                         | `Effect.catchTag("X", handler)`                 | Handle one tagged error                                         |
| `async function name(args)`                           | `Effect.fn("name")(function* (args) {})`        | Create a named effectful function                               |
| Add tracing                                           | `.pipe(Effect.withSpan("name"))`                | Add operational behavior outside the main logic                 |
| Add retry/timeout                                     | `.pipe(Effect.retry(...), Effect.timeout(...))` | Add policies around the program                                 |
| Simple transforms                                     | `.pipe(Effect.map(...))`                        | Transform the success value                                     |
| Retrieve dependencies                                 | `yield* Service` or service accessors           | Pull dependencies from the requirement channel                  |
| Constructor dependency injection                      | `Effect.Service` + `Layer`                      | Define and build dependencies                                   |
| Small local provide                                   | `program.pipe(Effect.provide(layer))`           | Provide dependencies to one program                             |
| Layer composition                                     | `Layer.mergeAll(...)` / `Layer.provide(...)`    | Build the dependency graph                                      |
| App container                                         | `ManagedRuntime`                                | Runtime with your dependency graph already provided             |
| Run the app                                           | `runtime.runPromise(program)`                   | Execute the lazy program at the application edge                |
