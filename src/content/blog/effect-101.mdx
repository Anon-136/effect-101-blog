# How To Read Effect Code For Beginner

People say Effect has a steep learning curve.

I get it.

When you first see `Effect.gen`, `yield*`, `Layer`, `Runtime`, `Data.TaggedError`, and services everywhere, it can feel like Effect is trying to replace TypeScript with a secret language.

I do not think that is the right way to read it.

Effect code is still TypeScript code.

The trick is learning how to read it.

Let me show you.

---

## Start Here: `Effect.gen`

This is the first Effect code you should learn to read.

```ts twoslash id="start-effect-gen"
import { Effect } from "effect";

interface User {
  readonly id: string;
  readonly name: string;
}

interface Appointment {
  readonly id: string;
  readonly userId: string;
  readonly status: "booked";
}

const getUser = () =>
  Effect.succeed<User>({
    id: "user_1",
    name: "Anon",
  });

const getAppointment = () =>
  Effect.succeed<Appointment>({
    id: "appointment_1",
    userId: "user_1",
    status: "booked",
  });

// ---cut---

const program = Effect.gen(function* () {
  const user = yield* getUser();
  const appointment = yield* getAppointment();

  console.log(`Loaded appointment for ${user.name}`);

  return appointment;
});

Effect.runPromise(program).then(console.log);

/*
Output:
Loaded appointment for Anon
{ id: "appointment_1", userId: "user_1", status: "booked" }
*/
```

Read this:

```ts id="read-yield"
const user = yield* getUser();
```

like this:

```ts id="read-await"
const user = await getUser();
```

That is the beginner mental model.

Inside `Effect.gen`:

```ts id="yield-meaning"
yield* someEffect;
```

means:

> Run this Effect step. If it succeeds, give me the success value. If it fails, stop the whole program with that failure.

So this:

```ts id="effect-gen-shape"
const program = Effect.gen(function* () {
  const user = yield* getUser();
  const appointment = yield* getAppointment();

  return appointment;
});
```

is similar to this normal TypeScript:

```ts id="async-await-shape"
async function program() {
  const user = await getUser();
  const appointment = await getAppointment();

  return appointment;
}
```

The shape is the same.

Effect is not asking you to forget `async/await`.

It is giving you a typed version of the same story.

## Normal `async/await` Version

Here is the same example in normal TypeScript.

```ts id="normal-async-await-version"
interface User {
  readonly id: string;
  readonly name: string;
}

interface Appointment {
  readonly id: string;
  readonly userId: string;
  readonly status: "booked";
}

const getUser = async (): Promise<User> => ({
  id: "user_1",
  name: "Anon",
});

const getAppointment = async (): Promise<Appointment> => ({
  id: "appointment_1",
  userId: "user_1",
  status: "booked",
});

async function program(): Promise<Appointment> {
  const user = await getUser();
  const appointment = await getAppointment();

  console.log(`Loaded appointment for ${user.name}`);

  return appointment;
}

program().then(console.log);

/*
Output:
Loaded appointment for Anon
{ id: "appointment_1", userId: "user_1", status: "booked" }
*/
```

This is readable.

The Effect version should be read almost the same way.

```ts id="yield-is-await"
const user = yield* getUser();
```

means:

```ts id="await-version"
const user = await getUser();
```

The difference is that `yield*` works with Effect values instead of Promise values.

## Make the Example Concrete: Typed Error Handling

Now let’s make the example more realistic.

We will model this flow:

1. Find a user.
2. Find the user's appointment.
3. Return the appointment.
4. If something fails, convert it into one domain error.

No dependency injection yet.

No service layer yet.

Just `Effect.gen`, `Data.TaggedError`, and error handling.

```ts twoslash id="concrete-error-handling"
import { Data, Effect } from "effect";

interface User {
  readonly id: string;
  readonly name: string;
}

interface Appointment {
  readonly id: string;
  readonly userId: string;
  readonly status: "booked" | "cancelled";
}

class UserNotFound extends Data.TaggedError("UserNotFound")<{
  readonly userId: string;
}> {}

class AppointmentNotFound extends Data.TaggedError("AppointmentNotFound")<{
  readonly userId: string;
}> {}

class BookingLookupError extends Data.TaggedError("BookingLookupError")<{
  readonly message: string;
  readonly cause: UserNotFound | AppointmentNotFound;
}> {}

const getUser = (userId: string) =>
  Effect.gen(function* () {
    if (userId !== "user_1") {
      return yield* new UserNotFound({ userId });
    }

    return {
      id: "user_1",
      name: "Anon",
    };
  });

const getAppointment = (userId: string) =>
  Effect.gen(function* () {
    if (userId !== "user_1") {
      return yield* new AppointmentNotFound({ userId });
    }

    return {
      id: "appointment_1",
      userId,
      status: "booked" as const,
    };
  });

const getUserAppointment = (userId: string) =>
  Effect.gen(function* () {
    const user = yield* getUser(userId);
    const appointment = yield* getAppointment(user.id);

    return appointment;
  }).pipe(
    Effect.mapError((error) => {
      switch (error._tag) {
        case "UserNotFound":
          return new BookingLookupError({
            message: `Cannot find user ${error.userId}`,
            cause: error,
          });

        case "AppointmentNotFound":
          return new BookingLookupError({
            message: `Cannot find appointment for user ${error.userId}`,
            cause: error,
          });
      }
    }),
  );

const program = getUserAppointment("user_1");

program;
// ^?

Effect.runPromise(program).then(console.log);

/*
Output:
{ id: "appointment_1", userId: "user_1", status: "booked" }
*/
```

Notice that we did not manually annotate the return type.

We let Effect infer it.

If you hover `program` in an editor, you get the story:

```ts id="program-hover"
Effect<Appointment, BookingLookupError, never>;
```

Read it as:

> This program succeeds with an `Appointment`, may fail with `BookingLookupError`, and needs no dependencies.

This is the value of Effect.

Normal TypeScript would likely say:

```ts id="promise-appointment"
Promise<Appointment>;
```

Effect tells you more:

```ts id="effect-appointment"
Effect<Appointment, BookingLookupError, never>;
```

That means the failure is not hidden.

It is part of the type.

## Compare to Normal `try/catch/throw`

Here is the same idea using normal TypeScript.

```ts id="normal-try-catch-throw"
interface User {
  readonly id: string;
  readonly name: string;
}

interface Appointment {
  readonly id: string;
  readonly userId: string;
  readonly status: "booked" | "cancelled";
}

class UserNotFound extends Error {
  constructor(readonly userId: string) {
    super(`User not found: ${userId}`);
  }
}

class AppointmentNotFound extends Error {
  constructor(readonly userId: string) {
    super(`Appointment not found for user: ${userId}`);
  }
}

class BookingLookupError extends Error {
  constructor(
    readonly message: string,
    readonly cause: unknown,
  ) {
    super(message);
  }
}

async function getUser(userId: string): Promise<User> {
  if (userId !== "user_1") {
    throw new UserNotFound(userId);
  }

  return {
    id: "user_1",
    name: "Anon",
  };
}

async function getAppointment(userId: string): Promise<Appointment> {
  if (userId !== "user_1") {
    throw new AppointmentNotFound(userId);
  }

  return {
    id: "appointment_1",
    userId,
    status: "booked",
  };
}

async function getUserAppointment(userId: string): Promise<Appointment> {
  try {
    const user = await getUser(userId);
    const appointment = await getAppointment(user.id);

    return appointment;
  } catch (error) {
    if (error instanceof UserNotFound) {
      throw new BookingLookupError(`Cannot find user ${error.userId}`, error);
    }

    if (error instanceof AppointmentNotFound) {
      throw new BookingLookupError(
        `Cannot find appointment for user ${error.userId}`,
        error,
      );
    }

    throw error;
  }
}

getUserAppointment("user_1").then(console.log);

/*
Output:
{ id: "appointment_1", userId: "user_1", status: "booked" }
*/
```

This works.

But the type still says:

```ts id="hidden-promise-type"
Promise<Appointment>;
```

The possible failures are invisible.

With Effect, the failure is in the type.

That is why Effect can feel more verbose at first.

It is not adding complexity for no reason.

It is moving hidden runtime behavior into the type system.

## A Simple Rule: `Effect.gen` for Logic, `.pipe()` for Composition

This is the style I recommend for beginners:

```ts id="gen-for-logic-pipe-for-composition-shape"
Effect.gen(function* () {
  // business logic lives here
})
.pipe(
 ... // composition happens here
);
```

Use `Effect.gen` for the main story.

Use `.pipe()` for behavior around the story.

Example:

```ts twoslash id="gen-for-logic-pipe-for-composition"
import { Data, Duration, Effect, Schedule } from "effect";

interface Post {
  readonly id: string;
  readonly title: string;
}

class DatabaseError extends Data.TaggedError("DatabaseError")<{
  readonly cause: unknown;
}> {}

const store = new Map<string, ReadonlyArray<Post>>();

const cacheGet = (key: string) =>
  Effect.gen(function* () {
    return store.get(key);
  });

const cacheSet = (key: string, value: ReadonlyArray<Post>) =>
  Effect.gen(function* () {
    store.set(key, value);
  });

const findPostsByUser = (userId: string) =>
  Effect.gen(function* () {
    if (userId === "broken") {
      return yield* new DatabaseError({
        cause: "database unavailable",
      });
    }

    return [
      {
        id: `post_${userId}`,
        title: "Effect makes hidden things visible",
      },
    ];
  });

const retryPolicy = Schedule.recurs(2).pipe(
  Schedule.addDelay(() => Duration.millis(100)),
);

const fetchUserPosts = (userId: string) =>
  Effect.gen(function* () {
    const cacheKey = `posts:${userId}`;

    const cached = yield* cacheGet(cacheKey);

    if (cached) {
      return cached;
    }

    const posts = yield* findPostsByUser(userId);

    yield* cacheSet(cacheKey, posts);

    return posts;
  }).pipe(
    Effect.withSpan("fetch_user_posts"),
    Effect.retry(retryPolicy),
    Effect.catchTag("DatabaseError", () => Effect.succeed([])),
  );

const program = fetchUserPosts("user_1");

Effect.runPromise(program).then(console.log);

/*
Output:
[
  {
    id: "post_user_1",
    title: "Effect makes hidden things visible"
  }
]
*/
```

Notice the shape:

```ts id="logic-and-composition-shape"
Effect.gen(function* () {
  const cacheKey = `posts:${userId}`;

  const cached = yield* cacheGet(cacheKey);

  if (cached) {
    return cached;
  }

  const posts = yield* findPostsByUser(userId);

  yield* cacheSet(cacheKey, posts);

  return posts;
}).pipe(
  Effect.withSpan("fetch_user_posts"),
  Effect.retry(retryPolicy),
  Effect.catchTag("DatabaseError", () => Effect.succeed([])),
);
```

Business logic is inside `Effect.gen`.

Operational behavior is inside `.pipe()`.

That separation makes the code easier to read.

## Don't Turn Everything Into a Pipeline Just Because It Is Fun

This is legal, but often harder to read:

```ts id="too-much-pipe"
Effect.succeed(order).pipe(
  Effect.andThen(validateOrder),
  Effect.andThen(calculateTotals),
  Effect.andThen(applyDiscounts),
  Effect.andThen(processPayment),
  Effect.andThen(sendConfirmation),
);
```

The problem is not that this is wrong.

The problem is that once you add conditionals, intermediate names, logs, or branching, this style becomes harder to scan.

Prefer this for sequential business logic:

```ts twoslash id="prefer-gen-over-too-much-pipe"
import { Effect } from "effect";

interface Order {
  readonly id: string;
  readonly total: number;
}

interface Payment {
  readonly orderId: string;
  readonly paid: true;
}

const order: Order = {
  id: "order_1",
  total: 100,
};

const validateOrder = (order: Order) =>
  Effect.gen(function* () {
    return order;
  });

const calculateTotals = (order: Order) =>
  Effect.gen(function* () {
    return {
      ...order,
      total: order.total + 10,
    };
  });

const applyDiscounts = (order: Order) =>
  Effect.gen(function* () {
    return {
      ...order,
      total: order.total - 20,
    };
  });

const processPayment = (order: Order) =>
  Effect.gen(function* () {
    return {
      orderId: order.id,
      paid: true as const,
    };
  });

const sendConfirmation = (payment: Payment) =>
  Effect.gen(function* () {
    console.log(`Confirmation sent for ${payment.orderId}`);

    return payment;
  });

const program = Effect.gen(function* () {
  const validated = yield* validateOrder(order);
  const withTotals = yield* calculateTotals(validated);
  const discounted = yield* applyDiscounts(withTotals);
  const payment = yield* processPayment(discounted);

  return yield* sendConfirmation(payment);
});

Effect.runPromise(program).then(console.log);

/*
Output:
Confirmation sent for order_1
{ orderId: "order_1", paid: true }
*/
```

My beginner decision matrix:

| Situation                            | Prefer       |
| ------------------------------------ | ------------ |
| Injecting or retrieving dependencies | `Effect.gen` |
| Conditional logic                    | `Effect.gen` |
| Sequential operations                | `Effect.gen` |
| Error handling                       | `.pipe()`    |
| Adding tracing                       | `.pipe()`    |
| Adding retry/timeout                 | `.pipe()`    |
| Layer building                       | `.pipe()`    |
| Simple transforms                    | `.pipe()`    |

Easy as that.

## Why `Effect.gen` Has No Arguments

This confuses many beginners.

You usually write:

```ts id="effect-gen-no-args-shape"
const program = Effect.gen(function* () {
  // ...
});
```

Not:

```ts id="effect-gen-with-args-wrong-shape"
const program = Effect.gen(function* (userId: string) {
  // ...
});
```

Why?

Because `Effect.gen` creates one lazy program.

It is not a function factory by itself.

This:

```ts twoslash id="effect-gen-no-args"
import { Effect } from "effect";

const program = Effect.gen(function* () {
  return 123;
});

Effect.runPromise(program).then(console.log);

/*
Output:
123
*/
```

means:

> Here is a program. It has no runtime arguments. Run it later.

If you need arguments, use a normal function that returns an Effect:

```ts twoslash id="normal-function-returning-effect"
import { Effect } from "effect";

const greet = (name: string) =>
  Effect.gen(function* () {
    return `Hello, ${name}`;
  });

const program = greet("Anon");

Effect.runPromise(program).then(console.log);

/*
Output:
Hello, Anon
*/
```

That is fine.

But for named business operations, prefer `Effect.fn`.

## `Effect.fn`: Named Effectful Functions

`Effect.fn` is for creating named functions that return Effects.

The arguments of `Effect.fn` should be normal business inputs:

- `userId`
- `appointmentId`
- `order`
- `paymentInput`
- `command`
- `query`

Dependencies should usually stay in the requirement channel.

Example:

```ts twoslash id="effect-fn-example"
import { Data, Effect } from "effect";

interface User {
  readonly id: string;
  readonly name: string;
}

interface Appointment {
  readonly id: string;
  readonly userId: string;
}

class UserNotFound extends Data.TaggedError("UserNotFound")<{
  readonly userId: string;
}> {}

class AppointmentNotFound extends Data.TaggedError("AppointmentNotFound")<{
  readonly userId: string;
}> {}

const getUser = (userId: string) =>
  Effect.gen(function* () {
    if (userId !== "user_1") {
      return yield* new UserNotFound({ userId });
    }

    return {
      id: userId,
      name: "Anon",
    };
  });

const getAppointment = (userId: string) =>
  Effect.gen(function* () {
    if (userId !== "user_1") {
      return yield* new AppointmentNotFound({ userId });
    }

    return {
      id: "appointment_1",
      userId,
    };
  });

const getAppointmentForUser = Effect.fn("getAppointmentForUser")(function* (
  userId: string,
) {
  const user = yield* getUser(userId);
  const appointment = yield* getAppointment(user.id);

  return appointment;
});

const program = getAppointmentForUser("user_1");

Effect.runPromise(program).then(console.log);

/*
Output:
{ id: "appointment_1", userId: "user_1" }
*/
```

Read this:

```ts id="effect-fn-read"
const getAppointmentForUser = Effect.fn("getAppointmentForUser")(function* (
  userId: string,
) {
  // ...
});
```

as:

> Define a named effectful function called `getAppointmentForUser`.

The name is useful for tracing, debugging, and observability.

The argument is the business input.

The function still returns a lazy Effect program.

## Effect Has Channels

Effect tracks different kinds of outcomes.

For beginners, think of three channels:

| Channel | Meaning                  | Example                                               |
| ------- | ------------------------ | ----------------------------------------------------- |
| Success | The program succeeded    | `Effect.succeed(value)`                               |
| Failure | Expected/tracked error   | `Effect.fail(error)` or `yield* new TaggedError(...)` |
| Defect  | Unexpected/untracked bug | `Effect.die(error)` or unhandled throw                |

Failure means:

> This is a normal error that can happen in the program, and I want it tracked.

Defect means:

> This should not happen, or I did not handle it as a typed failure.

---

## Failure: Expected Error

Use `Data.TaggedError` for expected domain errors.

Inside `Effect.gen`, tagged errors are yieldable.

```ts twoslash id="failure-channel-example"
import { Data, Effect } from "effect";

class InsufficientFunds extends Data.TaggedError("InsufficientFunds")<{
  readonly balance: number;
  readonly required: number;
}> {}

const chargeCard = () =>
  Effect.gen(function* () {
    return yield* new InsufficientFunds({
      balance: 50,
      required: 100,
    });
  });

const program = Effect.gen(function* () {
  return yield* chargeCard();
}).pipe(
  Effect.catchAll((error) =>
    Effect.succeed(
      `Payment failed: required ${error.required}, balance ${error.balance}`,
    ),
  ),
);

Effect.runPromise(program).then(console.log);

/*
Output:
Payment failed: required 100, balance 50
*/
```

The failure is tracked.

Before `catchAll`, the program may fail with `InsufficientFunds`.

After `catchAll`, the error is handled.

---

## Defect: Unexpected Error

Use `Effect.die` for bugs or impossible states.

```ts twoslash id="defect-channel-example"
import { Effect } from "effect";

const processPayment = (total: number) =>
  Effect.gen(function* () {
    if (total < 0) {
      return yield* Effect.die("Invariant violated: payment total is negative");
    }

    return {
      paid: true as const,
    };
  });

const program = processPayment(-1);

Effect.runPromiseExit(program).then(console.log);

/*
Output:
Exit.fail(...Cause.die...)
*/
```

A defect is not a normal business failure.

It means:

> Something is wrong with the program.

Examples:

- impossible state
- corrupted data
- programming mistake
- unexpected null
- broken invariant

Important:

```ts id="catch-all-does-not-catch-defect"
Effect.catchAll(...)
```

catches expected failures.

It does not handle defects like normal business errors.

That is good.

You usually do not want to recover from bugs in your domain flow.

---

## Throw and Reject

In normal JavaScript, anything can throw.

```ts id="throw-boom"
throw new Error("Boom");
```

In Effect, throwing inside Effect code is usually treated as an unexpected defect unless you deliberately capture it.

Use `Effect.try` or `Effect.tryPromise` when you want to convert thrown/rejected errors into the failure channel.

```ts twoslash id="try-convert-throw-to-failure"
import { Data, Effect } from "effect";

class ParseError extends Data.TaggedError("ParseError")<{
  readonly cause: unknown;
}> {}

const parseJson = (input: string) =>
  Effect.gen(function* () {
    return yield* Effect.try({
      try: () => JSON.parse(input) as unknown,
      catch: (cause) => new ParseError({ cause }),
    });
  });

const program = Effect.gen(function* () {
  const value = yield* parseJson("{ bad json }");

  return value;
}).pipe(
  Effect.catchAll((error) => {
    switch (error._tag) {
      case "ParseError":
        return Effect.succeed("Invalid JSON");
    }
  }),
);

Effect.runPromise(program).then(console.log);

/*
Output:
Invalid JSON
*/
```

Normal TypeScript version:

```ts id="normal-parse-json"
function parseJson(input: string): unknown {
  try {
    return JSON.parse(input);
  } catch {
    return "Invalid JSON";
  }
}

console.log(parseJson("{ bad json }"));

/*
Output:
Invalid JSON
*/
```

The normal version works.

But the type only says:

```ts id="unknown-type"
unknown;
```

The Effect version tracks the `ParseError` before you handle it.

Again, Effect makes the invisible part visible.

## Error Handling with `catchAll` and `switch`

Use `catchAll` when you want to handle every expected error in the failure channel.

```ts twoslash id="catch-all-switch-example"
import { Data, Effect } from "effect";

interface User {
  readonly id: string;
  readonly name: string;
}

class UserNotFound extends Data.TaggedError("UserNotFound")<{
  readonly userId: string;
}> {}

class AppointmentNotFound extends Data.TaggedError("AppointmentNotFound")<{
  readonly appointmentId: string;
}> {}

const getUser = (userId: string) =>
  Effect.gen(function* () {
    if (userId !== "user_1") {
      return yield* new UserNotFound({ userId });
    }

    return {
      id: userId,
      name: "Anon",
    };
  });

const getAppointment = (appointmentId: string) =>
  Effect.gen(function* () {
    if (appointmentId !== "appointment_1") {
      return yield* new AppointmentNotFound({ appointmentId });
    }

    return "Appointment found";
  });

const program = Effect.gen(function* () {
  const user = yield* getUser("user_1");
  const appointment = yield* getAppointment("missing");

  return `${appointment} for ${user.name}`;
}).pipe(
  Effect.catchAll((error) => {
    switch (error._tag) {
      case "UserNotFound":
        return Effect.succeed(`User missing: ${error.userId}`);

      case "AppointmentNotFound":
        return Effect.succeed(`Appointment missing: ${error.appointmentId}`);
    }
  }),
);

Effect.runPromise(program).then(console.log);

/*
Output:
Appointment missing: missing
*/
```

Normal `try/catch` catches thrown values.

Effect `catchAll` catches typed expected failures.

That is a major difference.

## Dependency Injection: The Requirement Channel

Now we can talk about dependencies.

In normal TypeScript, dependencies are usually passed as arguments or constructor parameters.

```ts id="normal-di"
class UserService {
  constructor(private readonly database: Database) {}
}
```

In Effect, dependencies can live in the requirement channel.

You do not need to manually pass them through every function.

Here is a small example.

```ts twoslash id="dependency-injection-requirement-channel"
import { Data, Effect } from "effect";

interface User {
  readonly id: string;
  readonly name: string;
}

class UserNotFound extends Data.TaggedError("UserNotFound")<{
  readonly userId: string;
}> {}

class UserService extends Effect.Service<UserService>()("UserService", {
  accessors: true,
  sync: () => ({
    getById: (userId: string) =>
      Effect.gen(function* () {
        if (userId !== "user_1") {
          return yield* new UserNotFound({ userId });
        }

        return {
          id: userId,
          name: "Anon",
        };
      }),
  }),
}) {}

const program = Effect.gen(function* () {
  const user = yield* UserService.getById("user_1");

  return user.name;
});

program;
// ^?

const runnable = program.pipe(Effect.provide(UserService.Default));

Effect.runPromise(runnable).then(console.log);

/*
Output:
Anon
*/
```

Before providing the service, the program needs `UserService`.

After this:

```ts id="provide-user-service"
program.pipe(Effect.provide(UserService.Default));
```

the requirement is gone.

Now the program is ready to run.

That is dependency injection in Effect.

The dependency is not passed as a function argument.

The dependency is tracked by Effect.

---

## Providing Dependencies Before Running

You cannot run a program that still needs requirements.

So you provide the requirement first.

For small examples, this is okay:

```ts id="effect-provide-small-example"
program.pipe(Effect.provide(UserService.Default));
```

For applications, prefer creating a runtime once.

```ts twoslash id="managed-runtime-example"
import { Data, Effect, ManagedRuntime } from "effect";

interface User {
  readonly id: string;
  readonly name: string;
}

class UserNotFound extends Data.TaggedError("UserNotFound")<{
  readonly userId: string;
}> {}

class UserService extends Effect.Service<UserService>()("UserService", {
  accessors: true,
  sync: () => ({
    getById: (userId: string) =>
      Effect.gen(function* () {
        if (userId !== "user_1") {
          return yield* new UserNotFound({ userId });
        }

        return {
          id: userId,
          name: "Anon",
        };
      }),
  }),
}) {}

const program = Effect.gen(function* () {
  return yield* UserService.getById("user_1");
});

const runtime = ManagedRuntime.make(UserService.Default);

runtime.runPromise(program).then(console.log);

/*
Output:
{ id: "user_1", name: "Anon" }
*/
```

This is the common app pattern:

1. Build services and layers once.
2. Create a `ManagedRuntime` once.
3. Use that runtime at the edge of the app.

For example:

- HTTP route handler
- queue worker
- CLI command
- cron job

Caution:

> Normally, do not create a new `ManagedRuntime` for every request.

Create it once at application startup, then reuse it.

## Services and Layers

A service describes what your app needs.

A layer describes how to build it.

```ts twoslash id="services-and-layers-example"
import { Data, Effect, Layer, ManagedRuntime } from "effect";

interface User {
  readonly id: string;
  readonly name: string;
}

interface Appointment {
  readonly id: string;
  readonly userId: string;
}

class UserNotFound extends Data.TaggedError("UserNotFound")<{
  readonly userId: string;
}> {}

class AppointmentNotFound extends Data.TaggedError("AppointmentNotFound")<{
  readonly userId: string;
}> {}

class UserService extends Effect.Service<UserService>()("UserService", {
  accessors: true,
  sync: () => ({
    getById: (userId: string) =>
      Effect.gen(function* () {
        if (userId !== "user_1") {
          return yield* new UserNotFound({ userId });
        }

        return {
          id: userId,
          name: "Anon",
        };
      }),
  }),
}) {}

class AppointmentService extends Effect.Service<AppointmentService>()(
  "AppointmentService",
  {
    accessors: true,
    sync: () => ({
      getByUserId: (userId: string) =>
        Effect.gen(function* () {
          if (userId !== "user_1") {
            return yield* new AppointmentNotFound({ userId });
          }

          return {
            id: "appointment_1",
            userId,
          };
        }),
    }),
  },
) {}

const MainLayer = Layer.mergeAll(
  UserService.Default,
  AppointmentService.Default,
);

const getAppointmentForUser = Effect.fn("getAppointmentForUser")(function* (
  userId: string,
) {
  const user = yield* UserService.getById(userId);
  const appointment = yield* AppointmentService.getByUserId(user.id);

  return appointment;
});

const runtime = ManagedRuntime.make(MainLayer);

runtime.runPromise(getAppointmentForUser("user_1")).then(console.log);

/*
Output:
{ id: "appointment_1", userId: "user_1" }
*/
```

Read the architecture like this:

```txt id="architecture-read"
Effect.fn
  -> named use case

Effect.gen
  -> sequential business logic

UserService / AppointmentService
  -> dependencies

Layer.mergeAll(...)
  -> dependency graph

ManagedRuntime.make(...)
  -> app runtime

runtime.runPromise(...)
  -> run at the edge
```

This is the point where Effect starts to feel like a framework, not just an error handling library.

## `Layer.provide` vs `Effect.provide`

Use `Effect.provide` when you are providing dependencies to a program.

```ts id="effect-provide"
program.pipe(Effect.provide(MainLayer));
```

Use `Layer.provide` when you are providing dependencies to another layer.

```ts id="layer-provide"
SomeLayer.pipe(Layer.provide(DependencyLayer));
```

Beginner rule:

| Situation                               | Use              |
| --------------------------------------- | ---------------- |
| I have a program and want to run it     | `Effect.provide` |
| I have a layer that needs another layer | `Layer.provide`  |

Most beginners will use `Effect.provide` first.

You can learn `Layer.provide` later when your dependency graph grows.

## Final Mental Model

Effect is a framework.

Its code is easiest to read when you start with `Effect.gen`.

`Effect.gen` lets you write business logic like `async/await`.

`yield*` is the Effect version of `await`.

`.pipe()` is for composition: error handling, tracing, retry, timeout, and providing layers.

`Effect.fn` is for named effectful functions with arguments.

`Data.TaggedError` is for typed domain errors.

The failure channel is for expected errors.

The defect channel is for bugs and impossible states.

The requirement channel is dependency injection.

`Layer` builds dependencies.

`ManagedRuntime` runs your app with those dependencies.

Once you understand that, Effect becomes much less scary.

It is not a secret language.

It is TypeScript with the hidden parts turned on.

## Beginner Cheatsheet: Effect vs Normal TypeScript

| Normal TypeScript                                     | Effect                                          | How to read it                                                   |
| ----------------------------------------------------- | ----------------------------------------------- | ---------------------------------------------------------------- |
| `Promise<A>`                                          | `Effect<A, E, R>`                               | A lazy program that can succeed, fail, and require dependencies  |
| `return value`                                        | `Effect.succeed(value)`                         | Create a successful Effect                                       |
| `throw error` for expected error                      | `yield* new TaggedError(...)`                   | Stop with a typed expected failure inside `Effect.gen`           |
| `throw error` for expected error outside `Effect.gen` | `Effect.fail(error)`                            | Create a failed Effect directly                                  |
| `throw error` for bug                                 | `Effect.die(error)`                             | Mark an impossible or unexpected defect                          |
| `async function name(args)`                           | `Effect.fn("name")(function* (args) {})`        | Create a named effectful function                                |
| `await promise`                                       | `yield* effect`                                 | Run this step and unwrap its success value                       |
| `try/catch`                                           | `Effect.catchAll`, `Effect.catchTag`            | Handle expected typed failures                                   |
| `catch (e) if e instanceof X`                         | `Effect.catchTag("X", handler)`                 | Handle one tagged error                                          |
| Sequential business logic                             | `Effect.gen(function* () {})`                   | Use this for the main story                                      |
| Conditional logic                                     | `Effect.gen(function* () {})`                   | Easier to read than a long pipeline                              |
| Retrieve dependencies                                 | `yield* Service` or service accessors           | Pull dependencies from the requirement channel                   |
| Error handling                                        | `.pipe(Effect.catchTag(...))`                   | Put composition around the main story                            |
| Add tracing                                           | `.pipe(Effect.withSpan("name"))`                | Add operational behavior outside the main logic                  |
| Add retry/timeout                                     | `.pipe(Effect.retry(...), Effect.timeout(...))` | Add policies around the program                                  |
| Simple transforms                                     | `.pipe(Effect.map(...))`                        | Transform the success value                                      |
| Too many `andThen`s                                   | Prefer `Effect.gen`                             | Sequential logic usually reads better with `yield*`              |
| Function argument dependency                          | Requirement channel                             | Dependencies can be tracked by Effect instead of passed manually |
| Constructor dependency injection                      | `Effect.Service` + `Layer`                      | Define and build dependencies                                    |
| App container                                         | `ManagedRuntime`                                | Runtime with your dependency graph already provided              |
| Run the app                                           | `runtime.runPromise(program)`                   | Execute the lazy program at the application edge                 |
| Small local provide                                   | `program.pipe(Effect.provide(layer))`           | Provide dependencies to one program                              |
| Layer composition                                     | `Layer.mergeAll(...)` / `Layer.provide(...)`    | Build the dependency graph                                       |
