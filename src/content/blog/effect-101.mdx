# Effect 101: How to Read Effect Code

Everyone warns you that [Effect.ts](https://effect.website/) has a steep learning curve. I felt it too. The first time I opened an Effect codebase and saw `Effect.gen`, `yield*`, `Layer`, `Runtime`, `Data.TaggedError`, and services scattered everywhere, my brain quietly went "nope." It looks like someone took TypeScript and replaced it with a secret language only the cool kids understand.

Here's the good news: **it isn't**. Effect code is still just TypeScript. The whole curve is really about learning how to _read_ it, and once a few patterns click, the scary parts stop being scary. So let me walk you through the ones that mattered most.

---

## `Effect.gen` Is Where It Clicks

If you only learn to read one piece of Effect code, make it this one. Everything else builds on top.

```ts twoslash id="start-effect-gen" collapse={1-30}
import { Effect } from "effect"

interface User {
  readonly id: string
  readonly name: string
}

interface Appointment {
  readonly id: string
  readonly userId: string
  readonly status: "booked"
}

const getUser = (): Effect.Effect<User> =>
  Effect.gen(function* () {
    return {
      id: "user_1",
      name: "John",
    }
  })

const getAppointment = (): Effect.Effect<Appointment> =>
  Effect.gen(function* () {
    return {
      id: "appointment_1",
      userId: "user_1",
      status: "booked",
    }
  })

const program = Effect.gen(function* () {
  const user = yield* getUser()
  const appointment = yield* getAppointment()

  console.log(`Loaded appointment for ${user.name}`)

  return appointment
})

Effect.runPromise(program).then(console.log)

/*
Output:
Loaded appointment for John
{ id: "appointment_1", userId: "user_1", status: "booked" }
*/
```

The mental model is honestly easier than it looks. Read this:

```ts id="read-yield"
const user = yield* getUser()
```

exactly like this:

```ts id="read-await"
const user = await getUser()
```

Inside `Effect.gen`, the line `yield* someEffect` is basically saying:

> Run this Effect step. \
> If it succeeds, give me the success value. \
> If it fails, stop the whole program with that failure.

And the Effect version above? It's telling the exact same story as this plain TypeScript:

```ts twoslash id="async-await-shape" collapse={1-22}
interface User {
  readonly id: string
  readonly name: string
}

interface Appointment {
  readonly id: string
  readonly userId: string
  readonly status: "booked"
}

const getUser = async (): Promise<User> => ({
  id: "user_1",
  name: "John",
})

const getAppointment = async (): Promise<Appointment> => ({
  id: "appointment_1",
  userId: "user_1",
  status: "booked",
})

async function program() {
  const user = await getUser()
  const appointment = await getAppointment()

  console.log(`Loaded appointment for ${user.name}`)

  return appointment
}

program().then(console.log)

/*
Output:
Loaded appointment for John
{ id: "appointment_1", userId: "user_1", status: "booked" }
*/
```

Same shape, line for line. Effect isn't asking you to throw away everything you know about `async/await`, it's just handing you a typed version of it. The only real difference: `yield*` works with Effect values, while `await` works with Promise values. That's it.

## Errors That Show Up in the Type

Let's make things a little more real. Here's a flow that finds a user, finds their appointment, returns it, and converts any failure into a single domain error.

```ts twoslash id="concrete-error-handling" collapse={1-53}
import { Data, Effect } from "effect"

interface User {
  readonly id: string
  readonly name: string
  readonly nextAppointmentId: string
}

interface Appointment {
  readonly id: string
  readonly status: "booked" | "cancelled"
}

class UserNotFound extends Data.TaggedError("UserNotFound")<{
  readonly userId: string
}> {}

class AppointmentNotFound extends Data.TaggedError("AppointmentNotFound")<{
  readonly appointmentId: string
}> {}

class BookingLookupError extends Data.TaggedError("BookingLookupError")<{
  readonly message: string
  readonly cause: UserNotFound | AppointmentNotFound
}> {}

const getUser = (userId: string): Effect.Effect<User, UserNotFound> =>
  Effect.gen(function* () {
    if (userId !== "user_1") {
      return yield* new UserNotFound({ userId })
    }

    return {
      id: userId,
      name: "John",
      nextAppointmentId: "appointment_1",
    }
  })

const getAppointment = (
  appointmentId: string,
): Effect.Effect<Appointment, AppointmentNotFound> =>
  Effect.gen(function* () {
    if (appointmentId !== "appointment_1") {
      return yield* new AppointmentNotFound({ appointmentId })
    }

    return {
      id: appointmentId,
      status: "booked",
    }
  })

const getUserAppointment = (userId: string) =>
  Effect.gen(function* () {
    const user = yield* getUser(userId)
    const appointment = yield* getAppointment(user.nextAppointmentId)

    return appointment
  }).pipe(
    Effect.catchTags({
      UserNotFound: (error) =>
        new BookingLookupError({
          message: `Cannot find user ${error.userId}`,
          cause: error,
        }),
      AppointmentNotFound: (error) =>
        new BookingLookupError({
          message: `Cannot find appointment ${error.appointmentId}`,
          cause: error,
        }),
    }),
  )

const program = getUserAppointment("user_1")
//    ^?

Effect.runPromise(program).then(console.log)

/*
Output:
{ id: "appointment_1", status: "booked" }
*/
```

Here's the neat part: notice we never wrote a return type. We let Effect infer it. Hover over `program` in your editor and the whole story is right there: `Effect<Appointment, BookingLookupError, never>`.
Read it left to right like this:

> This program succeeds with an `Appointment`, \
> may fail with `BookingLookupError`, \
> and needs no dependencies.

For comparison, here's the same idea done the old-fashioned way, with `try/catch`:

```ts twoslash id="normal-try-catch-throw" collapse={1-55}
interface User {
  readonly id: string
  readonly name: string
  readonly nextAppointmentId: string
}

interface Appointment {
  readonly id: string
  readonly status: "booked" | "cancelled"
}

class UserNotFound extends Error {
  constructor(readonly userId: string) {
    super(`User not found: ${userId}`)
  }
}

class AppointmentNotFound extends Error {
  constructor(readonly appointmentId: string) {
    super(`Appointment not found: ${appointmentId}`)
  }
}

class BookingLookupError extends Error {
  constructor(
    readonly message: string,
    readonly cause: unknown,
  ) {
    super(message)
  }
}

async function getUser(userId: string): Promise<User> {
  if (userId !== "user_1") {
    throw new UserNotFound(userId)
  }

  return {
    id: userId,
    name: "John",
    nextAppointmentId: "appointment_1",
  }
}

async function getAppointment(appointmentId: string): Promise<Appointment> {
  if (appointmentId !== "appointment_1") {
    throw new AppointmentNotFound(appointmentId)
  }

  return {
    id: appointmentId,
    status: "booked",
  }
}

async function getUserAppointment(userId: string): Promise<Appointment> {
  try {
    const user = await getUser(userId)
    const appointment = await getAppointment(user.nextAppointmentId)

    return appointment
  } catch (error) {
    if (error instanceof UserNotFound) {
      throw new BookingLookupError(`Cannot find user ${error.userId}`, error)
    }
    if (error instanceof AppointmentNotFound) {
      throw new BookingLookupError(
        `Cannot find appointment ${error.appointmentId}`,
        error,
      )
    }
    throw error
  }
}

const promise = getUserAppointment("user_1")
//    ^?

promise.then(console.log)

/*
Output:
{ id: "appointment_1", status: "booked" }
*/
```

This is the real value of Effect. Plain TypeScript would only tell you `Promise<Appointment>`. Effect tells you the fuller truth: `Effect<Appointment, BookingLookupError, never>`.
The failure isn't hidden, it's right there in the type.

## Wait, Why Does `Effect.gen` Take No Arguments?

This confuses a lot of beginners. You'll usually write:

```ts id="effect-gen-no-args-shape"
const program = Effect.gen(function* () {
  // ...
})
```

Not:

```ts id="effect-gen-with-args-wrong-shape"
const program = Effect.gen(function* (userId: string) {
  // ...
})
```

Why?

Because `Effect.gen` is purely for describing **computation**.
It's not the place to receive your business arguments.
The mental model that made it click for me:

> An `Effect` is a lazy computation. \
> It describes what should happen later. \
> It does not run immediately.

So this:

```ts id="effect-gen-no-args"
const program = Effect.gen(function* () {
  return 42
})
```

means:

> Here is a lazy program. \
> When someone runs it later, it will produce `42`.

`program` is not a function.

It's already an `Effect`.

It's a **result value** that **has not run yet**.

Think of `Effect.Effect<Success, Error>` as a `Promise<Result<Success, Error>>` but the promise hasn't run yet.

That's why `Effect.gen` doesn't need arguments here, there's nothing to pass in at runtime. The program already knows everything it needs to know.

But of course, most real business logic _does_ need inputs.

For example:

```ts id="business-operation-shape"
greet(name)
getUser(userId)
createAppointment(input)
```

In those cases, a normal function takes the arguments and then return an `Effect`.

```ts twoslash id="normal-function-returning-effect"
import { Effect } from "effect"

const greet = (name: string) =>
  Effect.gen(function* () {
    return `Hello, ${name}`
  })

greet
// ^?

const program = greet("John")
```

The outer function receives the argument.

```ts id="outer-function-receives-arguments"
const greet = (name: string) => ...
```

The inner `Effect.gen` describes the lazy computation.

```ts id="inner-effect-gen-describes-computation"
Effect.gen(function* () {
  return `Hello, ${name}`
})
```

So when you call `const program = greet("John")`, you're _not_ saying "**run this now**."

You're saying "Create a lazy Effect that already remembers `name = "John"`."

That `"John"` gets captured by the closure, and the Effect sits there patiently until you actually run it later.

```ts id="run-program-later"
Effect.runPromise(program)
```

This is the big difference from `async` functions.

```ts twoslash id="async-starts-immediately"
const greet = async (name: string) => {
  return `Hello, ${name}`
}

const promise = greet("John")

/*
Output:
Hello, John
*/
```

With `async`, the moment you call the function, the `Promise` is created _and already running eagerly_ .

That _eagerness_ is the important part.

Once a `Promise` starts, you're already too late to decide how it should run. You cannot first attach retry, timeout, logging, tracing, dependencies, or cancellation rules before the work begins.

`Effect` separates those two steps on purpose: the outer function receives the input, and `Effect.gen` describes the lazy program that will use that input later. Calling the function prepares the work. Running the Effect starts the work.

## `Effect.fn`: Give Your Functions a Name

`Effect.fn` lets you create named functions that return Effects. Its arguments should be normal business inputs you'd expect, things like `userId`, `appointmentId`, `order`, `paymentInput`, `command`, or `query`.

```ts twoslash id="effect-fn-example" collapse={1-44}
import { Data, Effect } from "effect"

interface User {
  readonly id: string
  readonly name: string
  readonly nextAppointmentId: string
}

interface Appointment {
  readonly id: string
}

class UserNotFound extends Data.TaggedError("UserNotFound")<{
  readonly userId: string
}> {}

class AppointmentNotFound extends Data.TaggedError("AppointmentNotFound")<{
  readonly appointmentId: string
}> {}

const getUser = Effect.fn("getUser")(function* (userId: string) {
  if (userId !== "user_1") {
    return yield* new UserNotFound({ userId })
  }
  const user: User = {
    id: userId,
    name: "John",
    nextAppointmentId: "appointment_1",
  }
  return user
})

const getAppointment = Effect.fn("getAppointment")(function* (
  appointmentId: string,
) {
  if (appointmentId !== "appointment_1") {
    return yield* new AppointmentNotFound({ appointmentId })
  }
  const appointment: Appointment = {
    id: appointmentId,
  }
  return appointment
})

const getAppointmentForUser = Effect.fn("getAppointmentForUser")(function* (
  userId: string,
) {
  const user = yield* getUser(userId)
  const appointment = yield* getAppointment(user.nextAppointmentId)

  return appointment
})

getAppointmentForUser
// ^?

const program = getAppointmentForUser("user_1")

Effect.runPromise(program).then(console.log)

/*
Output:
{ id: "appointment_1" }
*/
```

You can read `Effect.fn("getAppointmentForUser")` as:

> Define a named function called `getAppointmentForUser` that returns an Effect

That name isn't just decoration, it pays off in tracing, debugging, and observability. The argument is your business input, and the function still hands back a lazy Effect program, exactly like before.

## Three Channels to Keep in Mind

Effect keeps track of different kinds of outcomes, and as a beginner it helps to picture three channels:

| Channel | Meaning                  | Example                                               |
| ------- | ------------------------ | ----------------------------------------------------- |
| Success | The program succeeded    | `Effect.succeed(value)`                               |
| Failure | Expected/tracked error   | `Effect.fail(error)` or `yield* new TaggedError(...)` |
| Defect  | Unexpected/untracked bug | `Effect.die(error)` or unhandled throw                |

Think of it this way: a _failure_ is "this is a normal error that can happen, and I want it tracked." A _defect_ is "this should never happen, or I didn't want to handle it as a typed failure."

### Failure: Errors You Saw Coming

Use `Data.TaggedError` whenever you have an expected domain error. \
The bonus: inside `Effect.gen`, tagged errors are yieldable, so they slot right into your flow.

```ts twoslash id="failure-channel-example"
import { Data, Effect } from "effect"

class InsufficientFunds extends Data.TaggedError("InsufficientFunds")<{
  readonly balance: number
  readonly required: number
}> {}

const chargeCard = Effect.fn("chargeCard")(function* (balance: number) {
  const required = 100

  if (balance < required) {
    return yield* new InsufficientFunds({ balance, required })
  }

  return `Payment success: charged ${required}, balance ${balance}`
})

const program = (balance: number) =>
  chargeCard(balance).pipe(
    Effect.catchAll((error) =>
      Effect.succeed(
        `Payment failed: required ${error.required}, balance ${error.balance}`,
      ),
    ),
  )

Effect.runPromise(program(50)).then(console.log)
Effect.runPromise(program(150)).then(console.log)

/*
Output:
Payment failed: required 100, balance 50
Payment success: charged 100, balance 150
*/
```

See how the failure stays visible? Before `catchAll`, the program might fail with `InsufficientFunds`. After `catchAll`, that error is handled, so a low balance turns from failure into a success message while a sufficient remains succeed.

### Defect: Errors You Didn't

Use `Effect.die` when you hit a bug or an impossible state.

```ts twoslash id="defect-channel-example"
import { Effect } from "effect"

const processPayment = Effect.fn("processPayment")(function* (balance: number) {
  if (balance < 0) {
    return yield* Effect.die("Invariant violated: balance is negative")
  }

  return {
    paid: true,
  }
})

const program = processPayment(-1)

Effect.runPromiseExit(program).then(console.log)

/*
Output:
Exit.fail(...Cause.die...)
*/
```

A defect isn't your everyday business failure, it's a signal that something is genuinely wrong with the program itself: an impossible state, corrupted data, a programming mistake, a surprise `null`, a broken invariant. And here's a subtle but important detail, `Effect.catchAll(...)` scoops up expected failures but deliberately leaves defects alone. That's actually a feature, not a bug. You usually _don't_ want to quietly recover from real bugs in the middle of your business flow.

## `Effect.gen` for Logic, `.pipe()` for Everything Around It

If you're just starting out, here's the style I'd nudge you toward: let `Effect.gen` carry the main story, and let `.pipe()` handle the behavior wrapped around it.

```ts id="gen-for-logic-pipe-for-composition-shape"
Effect.gen(function* () {
 ... // business logic lives here
})
.pipe(
 ... // composition happens here
)
```

Here's a more complete example to see it in action:

```ts id="gen-for-logic-pipe-for-composition"
const fetchUserPosts = (userId: string) =>
  Effect.gen(function* () {
    const cacheKey = `posts:${userId}`
    const cached = yield* cacheGet(cacheKey)
    if (cached) {
      return cached
    }
    const posts = yield* findPostsByUser(userId)
    yield* cacheSet(cacheKey, posts)
    return posts
  }).pipe(
    Effect.withSpan("fetch_user_posts"),
    Effect.retry(retryPolicy),
    Effect.catchTag("DatabaseError", () => Effect.succeed([])),
  )
```

The business logic lives inside `Effect.gen`, while the operational stuff (`withSpan`, `retry`, `catchTag`) lives inside `.pipe()`. Keeping those two worlds apart is exactly what makes the code pleasant to read.

### Don't Pipe Everything Just Because You Can

The following is totally valid code, but it can be a slog to read:

```ts id="too-much-pipe"
Effect.succeed(order).pipe(
  Effect.andThen(validateOrder),
  Effect.andThen(calculateTotals),
  Effect.andThen(applyDiscounts),
  Effect.andThen(processPayment),
  Effect.andThen(sendConfirmation),
)
```

Nothing's _wrong_ with it. But the second you add a conditional, an intermediate variable, a log line, or a branch, this pipe-everything style gets hard to follow fast. For sequential business logic, I'd reach for this instead:

```ts id="prefer-gen-over-too-much-pipe"
const program = Effect.gen(function* () {
  const validated = yield* validateOrder(order)
  const withTotals = yield* calculateTotals(validated)
  const discounted = yield* applyDiscounts(withTotals)
  const payment = yield* processPayment(discounted)
  return yield* sendConfirmation(payment)
})
```

Here's my beginner decision matrix:

| Situation                            | Prefer       |
| ------------------------------------ | ------------ |
| Injecting or retrieving dependencies | `Effect.gen` |
| Conditional logic                    | `Effect.gen` |
| Sequential operations                | `Effect.gen` |
| Error handling                       | `.pipe()`    |
| Adding tracing                       | `.pipe()`    |
| Adding retry/timeout                 | `.pipe()`    |
| Layer building                       | `.pipe()`    |
| Simple transforms                    | `.pipe()`    |

[Thanks to Dillon](https://x.com/dillon_mulroy/status/1936530534936486009?s=20)

## Dependencies Without the Plumbing

Now we can talk about dependencies. In everyday TypeScript, you usually pass dependencies around as function arguments or constructor parameters.

```ts id="normal-di"
class UserService {
  constructor(private readonly database: Database) {}
}
```

In Effect, dependencies can instead live in the requirement channel, which means you don't have to manually thread them through every single function. Here's a small example.

```ts twoslash id="dependency-injection-requirement-channel" collapse={1-30}
import { Data, Effect } from "effect"

interface User {
  readonly id: string
  readonly name: string
}

class UserNotFound extends Data.TaggedError("UserNotFound")<{
  readonly userId: string
}> {}

class UserService extends Effect.Service<UserService>()("UserService", {
  effect: Effect.sync(() => {
    const getById = Effect.fn("getById")(function* (userId: string) {
      if (userId !== "user_1") {
        return yield* new UserNotFound({ userId })
      }

      const user: User = {
        id: userId,
        name: "John",
      }

      return user
    })

    return { getById }
  }),
}) {}

const program = Effect.gen(function* () {
  const userService = yield* UserService
  const user = yield* userService.getById("user_1")

  return user
})

program
// ^?

const runnable = program.pipe(Effect.provide(UserService.Default))

runnable
// ^?

Effect.runPromise(runnable).then(console.log)

/*
Output:
{ id: "user_1", name: "John" }
*/
```

Before you provide the service, the program is still waiting on `UserService` in program.

But in runnable, the requirement vanishes and the program is ready to run. That's dependency injection in Effect: the dependency isn't passed in as a function argument, Effect itself keeps track of it for you.

## Wiring It Up Before You Run

A program that still needs requirements can't run, so you have to provide them first. For small examples, this does the job nicely:

```ts id="effect-provide-small-example"
program.pipe(Effect.provide(UserService.Default))
```

For real applications, though, you'll want to build a runtime once and reuse it.

```ts twoslash id="managed-runtime-example" collapse={1-34}
import { Data, Effect, ManagedRuntime } from "effect"

interface User {
  readonly id: string
  readonly name: string
}

class UserNotFound extends Data.TaggedError("UserNotFound")<{
  readonly userId: string
}> {}

class UserService extends Effect.Service<UserService>()("UserService", {
  accessors: true,
  effect: Effect.sync(() => {
    const getById = Effect.fn("getById")(function* (userId: string) {
      if (userId !== "user_1") {
        return yield* new UserNotFound({ userId })
      }

      const user: User = {
        id: userId,
        name: "John",
      }

      return user
    })

    return { getById }
  }),
}) {}

const program = Effect.gen(function* () {
  return yield* UserService.getById("user_1")
})

const runtime = ManagedRuntime.make(UserService.Default)

runtime.runPromise(program).then(console.log)

/*
Output:
{ id: "user_1", name: "John" }
*/
```

The usual app pattern looks like this: build your services and layers once, create a `ManagedRuntime` once, and then use that runtime at the edge of your app, such as an HTTP route handler, a queue worker, a CLI command, or a cron job, wherever the outside world meets your code.

> Do not `Effect.provide` services for every request. \
> create a new `ManagedRuntime` once when your app boots, then reuse it everywhere.

## Services Say What, Layers Say How

A service describes _what_ your app needs, and a layer describes _how_ to build it.

```ts twoslash id="services-and-layers-example" collapse={1-63}
import { Data, Effect, Layer, ManagedRuntime } from "effect"

interface User {
  readonly id: string
  readonly name: string
}

interface Appointment {
  readonly id: string
  readonly userId: string
}

class UserNotFound extends Data.TaggedError("UserNotFound")<{
  readonly userId: string
}> {}

class AppointmentNotFound extends Data.TaggedError("AppointmentNotFound")<{
  readonly userId: string
}> {}

class UserService extends Effect.Service<UserService>()("UserService", {
  accessors: true,
  effect: Effect.sync(() => {
    const getById = Effect.fn("getById")(function* (userId: string) {
      if (userId !== "user_1") {
        return yield* new UserNotFound({ userId })
      }

      const user: User = {
        id: userId,
        name: "John",
      }

      return user
    })

    return { getById }
  }),
}) {}

class AppointmentService extends Effect.Service<AppointmentService>()(
  "AppointmentService",
  {
    accessors: true,
    effect: Effect.sync(() => {
      const getByUserId = Effect.fn("getByUserId")(function* (userId: string) {
        if (userId !== "user_1") {
          return yield* new AppointmentNotFound({ userId })
        }

        const appointment: Appointment = {
          id: "appointment_1",
          userId,
        }

        return appointment
      })

      return { getByUserId }
    }),
  },
) {}

const MainLayer = Layer.mergeAll(
  UserService.Default,
  AppointmentService.Default,
)

const getAppointmentForUser = Effect.fn("getAppointmentForUser")(function* (
  userId: string,
) {
  const user = yield* UserService.getById(userId)
  const appointment = yield* AppointmentService.getByUserId(user.id)

  return appointment
})

const runtime = ManagedRuntime.make(MainLayer)

runtime.runPromise(getAppointmentForUser("user_1")).then(console.log)

/*
Output:
{ id: "appointment_1", userId: "user_1" }
*/
```

Once it's wired up, you can read the whole architecture straight down the page:

```txt id="architecture-read"
Effect.fn
  -> named use case

UserService / AppointmentService
  -> dependencies

Layer.mergeAll(...)
  -> dependency graph

ManagedRuntime.make(...)
  -> app runtime

runtime.runPromise(...)
  -> run at the edge
```

This is the moment Effect stops feeling like "an error-handling library" and starts feeling like a real framework.

## `Layer.provide` or `Effect.provide`?

Use `Effect.provide` when you're providing dependencies to a program:

```ts id="effect-provide"
program.pipe(Effect.provide(MainLayer))
```

Use `Layer.provide` when you're providing dependencies to _another layer_:

```ts id="layer-provide"
UserService.pipe(Layer.provide(Database))
```

A simple beginner rule:

| Situation                               | Use              |
| --------------------------------------- | ---------------- |
| I have a program and want to run it     | `Effect.provide` |
| I have a layer that needs another layer | `Layer.provide`  |

Most beginners reach for `Effect.provide` first, and you can pick up `Layer.provide` later when your dependency graph grows.

## Cheatsheet: Effect vs Normal TypeScript

| Normal TypeScript                                     | Effect                                          | How to read it                                                  |
| ----------------------------------------------------- | ----------------------------------------------- | --------------------------------------------------------------- |
| `Promise<A>`                                          | `Effect<A, E, R>`                               | A lazy program that can succeed, fail, and require dependencies |
| `await promise`                                       | `yield* effect`                                 | Run this step and unwrap its success value                      |
| Business logic                                        | `Effect.gen(function* () {})`                   | Easier to read than a long pipeline                             |
| `return value`                                        | `Effect.succeed(value)`                         | Create a successful Effect                                      |
| `throw error` for expected error                      | `yield* new TaggedError(...)`                   | Stop with a typed expected failure inside `Effect.gen`          |
| `throw error` for expected error outside `Effect.gen` | `Effect.fail(error)`                            | Create a failed Effect directly                                 |
| `throw error` for bug                                 | `Effect.die(error)`                             | Mark an impossible or unexpected defect                         |
| `try/catch`                                           | `Effect.catchAll`, `Effect.catchTag`            | Handle expected typed failures                                  |
| `catch (e) if e instanceof X`                         | `Effect.catchTag("X", handler)`                 | Handle one tagged error                                         |
| `async function name(args)`                           | `Effect.fn("name")(function* (args) {})`        | Create a named effectful function                               |
| Add tracing                                           | `.pipe(Effect.withSpan("name"))`                | Add operational behavior outside the main logic                 |
| Add retry/timeout                                     | `.pipe(Effect.retry(...), Effect.timeout(...))` | Add policies around the program                                 |
| Simple transforms                                     | `.pipe(Effect.map(...))`                        | Transform the success value                                     |
| Retrieve dependencies                                 | `yield* Service` or service accessors           | Pull dependencies from the requirement channel                  |
| Constructor dependency injection                      | `Effect.Service` + `Layer`                      | Define and build dependencies                                   |
| Small local provide                                   | `program.pipe(Effect.provide(layer))`           | Provide dependencies to one program                             |
| Layer composition                                     | `Layer.mergeAll(...)` / `Layer.provide(...)`    | Build the dependency graph                                      |
| App container                                         | `ManagedRuntime`                                | Runtime with your dependency graph already provided             |
| Run the app                                           | `runtime.runPromise(program)`                   | Execute the lazy program at the application edge                |
